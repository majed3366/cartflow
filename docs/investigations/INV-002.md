## Observed Symptoms

1. Simulation wrote **27 carts**, **20 Purchase Truth**, reasons, schedules, mock WA under store slug **`demo`**.
2. Merchant walkthrough used a newly signed-up store (`متجر-واقع-صغير-*`) — **different identity**.
3. Home / Carts / dashboard APIs for that merchant session show **empty** queues / `no_carts` activation.
4. Home screenshots show “CartFlow يجهّز فهم متجرك…” skeleton.
5. Checkpoint V2 service probes on `demo` + sim QTC show rich Knowledge — **not** the merchant session path.
6. Phase 3.1 closed a **write-path** escape; read-path merchant binding remains unsolved for lab/demo.

## Root Cause Analysis

**Confirmed (2026-07-16):**

1. Canonical simulation target is `demo`.
2. Signup creates a distinct Store and sets `MerchantUser.primary_store_id` to it.
3. Dashboard / Home / session Knowledge / session Brief resolve store from `primary_store_id` → signup slug.
4. Reality Lab bind attempts `user.store_id` (nonexistent) and never reassigns `primary_store_id` to `demo`.
5. Checkpoint “surfaces see data” used hardcoded `demo` service probes under QTC — orthogonal to session UI.

**Root cause statement:** There is no governed **merchant session → store_slug → simulation-run** identity contract for lab/demo visibility. Session store and truth store diverge without explicit remapping or labelled dual-read policy.

## Merchant Impact

- “I ran reality and my dashboard is empty” — catastrophic trust failure.
- Merchant cannot distinguish “wrong store” from “product broken.”
- Contaminates every child surface until identity is aligned or explicitly labelled.

---

## Architectural Impact

- Store slug is tenant key and experience key without a sim/lab binding layer.
- Write isolation ≠ read visibility.
- Children INV-003/004/005/007 cannot close honestly while session ≠ truth store.

---

## Risk

| Horizon | Risk |
|---------|------|
| **Current** | Lab cannot validate merchant trust; false negatives on all surfaces |
| **Future** | Multi-store, agency, demo-share scenarios will hide or leak truth without a contract |

---

## Possible Solutions

*(Listed for Board context only — **not** an implementation authorization.)*

1. Lab merchant binding — review sessions explicitly bound to `demo`.
2. Simulation run claim — attach `simulation_run_id` → store for reads.
3. Store switcher honesty — never silent empty.
4. Governed lab-only read-through policy (must not break write isolation).
5. Separate Reality Review surface (ops/lab), not merchant product.
6. Process constraint until product contract exists.

---

## Recommended Direction

Board selects Implementation Architecture after DoR. Prefer explicit attach + visible store context for product truth; process constraint for lab until then. Never loosen Phase 3.1 write isolation while solving reads.
