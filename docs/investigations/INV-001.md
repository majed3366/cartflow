## Observed Symptoms

1. Historical simulation wrote durable reality dated **2026-05-01 → 2026-05-03**.
2. Merchant review ran on wall calendar **2026-07-15**.
3. Knowledge / dashboard KPIs for “last 7 days” (wall clock) report **zero carts / zero purchases** for `demo`.
4. The same Knowledge path, when evaluated with `now` at simulation window end, produces **rich, evidence-backed insights**.
5. Date chrome on merchant surfaces shows wall July, with **no cue** that lived history is in May.
6. Hybrid timestamps observed: May `first_seen` vs July `last_seen` after purchase reconcile (Lab T9).

## Root Cause Analysis

**Not assumptions — grounded in lab contrast:**

1. **Write path (simulation):** Store Reality Simulator injects events under SimulationClock; carts / Purchase Truth / reasons materialise with simulated timestamps (May 2026).
2. **Read path (merchant surfaces):** Knowledge and dashboard summary evaluate eligibility and metrics against **wall-clock** “now” and rolling windows (e.g. last 7 days from July 15 → window excludes May).
3. **Proof of causation:** Identical store (`demo`) + identical durable rows → empty Knowledge at wall `now`; populated Knowledge at sim-end `now`. The variable that flips the outcome is **time authority on the query**, not absence of data.
4. **Secondary drift:** Purchase reconcile (and similar post-paths) can stamp `last_seen_at` with wall clock while `first_seen` remains simulated → hybrid timeline rows (Lab architectural finding #3).
5. **Ownership gap:** There is no single merchant-visible **time contract** declaring which clock governs KPIs, Knowledge windows, Attention eligibility, and UI date chrome during / after historical simulation (or for any backdated truth).

**Root cause (statement):** CartFlow lacks a unified **time authority** for merchant read models relative to durable event time; wall-clock windows silently drop valid historical truth.

**Review validation (2026-07-15):** **Confirmed** — see [INV-001_REVIEW.md](./INV-001_REVIEW.md). Controlled KL contrast (`now` wall vs sim-end) + `_CLOCK_PATCH_MODULES` write-only inventory + unpatched `main` reconcile wall `recovered_at`.

- Monthly Summary (windowed materialisation — see INV-007)

---

## Merchant Impact

- Merchant hears “you have no carts / insufficient data” while the store **did** live three days of reality.
- Trust collapses: product appears broken or lying, not “out of window.”
- No UI language explains wrong clock vs empty store vs warming state.
- Reduces confidence that CartFlow “watched” the store.

---

## Architectural Impact

- SimulationClock is incomplete if only writers respect it.
- Read models become **non-deterministic relative to truth age** — same DB, different answers by wall date.
- Prevents honest historical replay, audits, and lab verification.
- Contaminates child investigations (Knowledge empty looks like Knowledge bugs; Attention empty looks like UX bugs).

---

## Possible Solutions

*(Options only — do not implement in this phase.)*

1. **Single Time Authority service** — all merchant read windows take an explicit `as_of` / clock source (wall | simulation | store-local).
2. **Durable event-time windows** — KPIs/Knowledge default to event timestamps with declared window, not wall “last 7 days” alone.
3. **Merchant-visible time contract** — UI always shows the window and clock in use; never imply “today” when querying history.
4. **Forbid wall mutation of simulated timestamps** — reconcile/update paths must preserve SimulationClock when a run is active / for tagged rows.
5. **Dual reporting** — “Operating window (wall)” vs “Lived history (event time)” as separate surfaces.
6. **Lab-only bridge** — force dashboard session `as_of=sim_end` when viewing demo after a run (mitigation, not product truth).

---

## Recommended Direction

Pursue **(1) + (2) + (3)**: establish one time-authority contract for reads, key windows to event time (or explicit as_of), and make the window visible to merchants. Treat (6) as lab tooling only, never as the product fix.

**Why:** Lab proof shows data is present; the failure is authority and honesty of the clock, not missing pipelines. Child Knowledge/Attention work will misdiagnose without this.

---

## Verification Plan

1. Re-run or reuse Small Reality May window on `demo`.
2. Query Knowledge + dashboard summary at wall `now` **and** at `as_of=sim_end` with the same store.
3. Assert: with correct time authority, both paths either show the same lived metrics **or** wall path explicitly labels “outside window” with non-zero historical totals available via as_of.
4. Assert: no hybrid May/July stamps on tagged simulation carts after reconcile under the chosen contract.
5. Screenshot + JSON evidence pack attached to case closure.
