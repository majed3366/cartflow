# CartFlow — إرشاد تشغيلي قابل للتنفيذ (مسارات)

## الفلسفة

طبقة `cartflow_admin_action_guidance` لا تحسب صحة جديدة؛ تترجم مخرجات **`build_admin_operational_summary_readonly()`** إلى:

- عنوان واضح  
- معنى تشغيلي  
- إجراء مقترح  
- **مسار داخلي اختياري** (روابط/مراسي جاهزة لصفحات لاحقة)  

الهدف: المشغّل يعرف **ماذا يفعل** و**إلى أين ينتقل** دون خبرة عميقة في النظام.

## مبادئ توجيه الإجراءات

1. **التحفيز لا الأوامر** — صياغة هادئة، بلا لغة إنذار مبالغ فيها.  
2. **أولوية فقط** — تُبنى قائمة الإجراءات عندما تبرر الأولوية أو إشارات شديدة (`derive_actionable_operational_items` + `_allow_action_cards`).  
3. **لا ضجيج** — في الوضع المستقر لا تُعرض بطاقات فارغة مزعجة؛ تُعرض رسالة «لا إجراءات مطلوبة» بثقة.  
4. **روابط آمنة** — المسارات معرفة كنص ثابت (مثل `/admin/runtime#duplicates`) — لا معلمات سرية، لا `POST` من الروابط.

## جاهزية التنقل الإداري

الصفحات الهدف **قد لا تكون مبنية بعد**؛ الغرض هو تثبيت هيكل مسارات الإدارة مبكراً:

| مسار تقريبي | غرض تشغيلي |
|-------------|-------------|
| `/admin/settings/whatsapp` | إعدادات مزود واتساب |
| `/admin/settings/production` | الإنتاج مقابل Sandbox |
| `/admin/onboarding` | تهيئة المتاجر والالتقاط |
| `/admin/runtime` | صحة تشغيل وتكرار ودورة حياة |
| `/admin/providers` | مركز مزودين (للتوسع لاحقاً) |

المراسي (`#overview`, `#duplicates`, …) تساعد في التمرير عندما تكون الصفحة متعددة الأقسام.

## قواعد الهدوء في الواجهة

- أزرار/روابط صغيرة، حدود خفيفة، بدون بانرات ضخمة.  
- «ما الذي يعنيه هذا؟» و «ما الإجراء المقترح؟» لتوضيح المعنى دون تكرار فني.

## التوسع المستقبلي

- ربط كل مسار بصفحة إعداد حقيقية دون تغيير منطق الملخص.  
- إضافة رموز حالة بصرية خفيفة لكل `code` إذا لزم.  
- ترجمة إضافية أو LTR عند الحاجة.

## الملفات

- `services/cartflow_admin_action_guidance.py` — الاشتقاق الحتمي.  
- `templates/admin_operations.html` — قسم «٢ب — مسارات إجراء مقترحة» وربط التحذيرات بالمرسى.  
- `routes/admin_operations.py` — تمرير `action_items` و `action_meta`.
